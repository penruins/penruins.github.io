3.1 分布式文件系统HDFS简介

全称Hadoop Distributed File System

大数据的两大核心技术：分布式存储、分布式处理

HDFS就是为了解决分布式存储的问题

HDFS实现目标

  • 兼容廉价的硬件设备
  • 实现流数据读写（传统的文件系统通常是以块为单位）
  • 支持大数据集
  • 支持简单的文件模型
  • 强大的跨平台兼容性（Hadoop是用java开发，java有很好的跨平台特性）

HDFS自身的局限性

  • 不适合低延迟数据访问（不能满足实时的处理需求）
  • 无法高效存储大量小文件
  • 不支持多用户写入及任意修改文件（只允许追加，不允许修改）

3.2 HDFS相关概念

HDFS当中最核心的概念：块

和普通的块的联系：为了分摊磁盘读写开销，也就是在大量数据间分摊磁盘寻址的开销

区别：HDFS的一个块要比普通文件系统的块大很多（普通的块可能是几千个字节，HDFS的块大小64M或128M

为什么要这样设计？

  • 支持面向大数据数据存储
  • 降低分布式节点的寻址开销

缺点：如果块过大会导致MapReduce就一两个任务在执行完全牺牲了MapReduce的并行度，发挥不了分布式并行处理的效果

HDFS两大组件：

​ 名称节点（主节点）：整个HDFS集群的管家，相当于是一个数据目录

​ 数据结点（DataNode）：存储实际的数据

元数据

​ 文件是什么

​ 文件被分成多少块

​ 每个块和文件是怎么映射的

​ 每个块被存储在哪个服务器上面

名称节点：

​ FSImage: 保存系统文件树（文件的复制登记、修改和访问时间、访问权限、块大小以及组成文件的块）

​ EditLog: 记录对数据进行的诸如创建、删除、重命名等操作

注：之前说名称节点会保存块在哪个数据结点存储，但这些数据并不在FSImage中存储，而是单独在内存的一个区域维护的

3.3 HDFS体系结构

HDFS命名空间：只有一个，它包含目录、文件、块

​ /+ 目录名称 例如：/user/shares

局限性：

​ 命名空间限制：名称节点是保存在内存中的，因此，名称节点能够容纳的对象（文件、块）的个数会受到空间大小限制

​ 性能的瓶颈：整个分布式文件的吞吐量，受限于单个名称节点的吞吐量

​ 隔离问题：由于集群中只有一个名称节点，只有一个命名空间，因此无法对不同应用程序进行隔离

​ 集群的可用性：一旦这个唯一的名称节点发生故障，会导致整个集群变得不可用

热备份：一旦发生故障马上恢复

冷备份：第一个发生故障后，必须停止一段时间，慢慢恢复

3.4 HDFS存储原理

伪分布式：把名称节点和数据结点都放在同一台机子上

冗余存储的好处：

  • 加快数据传输速度
  • 很容易检查数据错误
  • 保证数据可靠性

数据读取的原则：就近读取

​ HDFS提供了一个API可以确定一个数据节点所属的机架ID，客户端也可以调用API获取自己所属的机架ID

​ 当客户端读取数据时，从名称节点获得数据块不同副本的存放位置列表，列表中包含了副本所在的数据节点，可以调用API来确定客户端和这些数据结点所属的机架ID，当发现某个数据块副本对应的机架ID和客户端对应的机架ID相同时就优先选择该副本读取数据，如果没有发现，就随机选择一个副本读取数据

数据的错误与恢复

​ 名称节点错误：冷备份到第二名称节点，当出现错误时会暂停服务一段时间，从第二名称节点上恢复过来

​ 数据节点出错：数据结点会定时向名称节点发送心跳信息高速名称节点我还活着，当名称节点没收到时就断定数据结点发生了故障

​ 数据本身出错：校验码

HDFS和其他分布式文件管理系统最大的区别：HDFS可以调整冗余数据的位置

7.1.1分布式并行编程

MapReduce是一种分布式并行编程框架

摩尔定律：CPU每隔18个月性能就可以翻一番 05年开始失效

​ 单位面积内能集成的晶体管的数量是有上限的

数据处理能力提升的两条路线

​ 单核CPU到双核到四核到八核

​ 分布式并行编程（借助一个集群通过多台机器同时并行处理大规模数据集）

相关的并行编程框架：

​ MPI（消息传递接口，一种非常典型的并行编程框架）

​ OpenCL

​ CUDA

并行编程模型和传统的并行编程框架的区别

​ 传统：共享式架构（容错性差）、刀片服务器（价格高、拓展性差）、编程难度高（做什么+怎么做）、实时的细粒度计算

​ MapReduce：非共享式架构（容错性好）、扩展性好、编程难度低（做什么）、非实时的批处理和数据密集型的应用

7.1.2 MapReduce模型简介

MapReduce两个函数

​ Map

​ Reduce

策略

​ 分而治之：被非常庞大的数据集，切分成非常多的独立的小分片，然后为每一个分配单独地启动一个map任务，最终通过多个map任务，并行地在多个机器上去处理

理念

​ 计算向数据靠拢（将应用程序分发到数据所在的机器）而不是数据向计算靠拢（将数据拉到应用程序所在的机器）

Master/slave架构

​ 一个Master服务器：作业跟踪器JobTracker(负责整个作业的调度和处理以及失败和恢复)

​ 若干个slave服务器：负责具体任务执行的组件TaskTracker（负责接收JobTracker给它发的作业处理指令，完成具体的任务处理）

Map 键值对

Reduce

​ 输入：键值对列表

​ 输出：键值对

7.2 MapReduce的体系结构

Client

​ 通过Client可以提交用户编写的应用程序，用户通过它将应用程序教导JobTracker端

​ 通过这些Client用户也可以通过它提供的一些接口去查看当前提交作业的运行状态

JobTracker

​ 负责资源的监控和作业的调度

​ 监控底层的其它的TaskTracker以及当前运行的Job的健康状况

​ 一旦探测到失败的情况就把这个任务转移到其他节点继续执行，跟踪任务执行进度和资源使用量

TaskScheduler

TaskTracker

​ 执行具体的相关任务，一般接收JobTracker发送过来的命令

​ 把一些自己的资源使用情况，以及任务的运行进度通过心跳的方式发送给JobTracker

slot

​ map类型的slot

​ reduce类型的slot

Task

​ map任务 map函数

​ reduce任务 reduce函数

7.3 MapReduce工作流程

输入->Map任务->Reduce任务->输出

Reduce任务的数量

​ 最优的Reduce任务个数取决于集群中可用的reduce任务槽(slot)的数目

​ 通常设置比reduce任务槽数目稍微小一些的Reduce任务个数（这样可以预留一些系统资源处理可能发生的错误）

7.4 Shuffle过程原理

map键值对先写入缓存->溢写(分区、排序、合并)->磁盘文件归并

之后数据被Reduce任务取走->归并->reduce函数->输出

shuffle过程

  1. map端shuffle
    1. 输入数据和执行Map任务
    2. 写入缓存
    3. 溢写（分区、排序、合并） 溢写比例0.8
    4. 文件归并
  2. reduce端shuffle
