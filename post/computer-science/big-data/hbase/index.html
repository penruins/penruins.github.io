    hbase#

    • HBase 是一个面向列式存储的分布式数据库，其设计思想来源于 Google 的 BigTable 论文
    • HBase 底层存储基于 HDFS 实现，集群的管理基于 ZooKeeper 实现
    • HBase 良好的分布式架构设计为海量数据的快速存储、随机访问提供了可能，基于数据副本机制和分区机制可以轻松实现在线扩容、缩容和数据容灾，是大数据领域中 Key-Value 数据结构存储最常用的数据库方案

    hbase vs elasticsearch#

    • HBase是基于Hadoop生态的分布式列式数据库，源自Google BigTable设计，擅长海量结构化/半结构化数据的存储和高吞吐写入
    • Elasticsearch是基于Lucene的分布式搜索引擎，专注于全文检索、复杂查询和实时分析，采用文档型数据模型
    • 数据模型与灵活性
      • HBase采用列式存储模型，数据按"行键(RowKey)+列族(Column Family)+列限定符(Qualifier)“组织，适合结构化或半结构化数据。虽然schema-free设计允许灵活添加列，但列族需要预先定义
      • Elasticsearch采用文档型模型，数据以JSON格式存储，天然支持非结构化数据。字段可以动态添加，无需预定义schema，更适合多变的数据结构
    • 查询能力对比
      • HBase查询能力相对有限
        • 主要支持基于RowKey的精确查询和范围扫描(Scan)
        • 复杂查询需要借助二级索引或Phoenix等SQL层
        • 无内置全文检索能力，需结合Solr等搜索引擎
      • Elasticsearch提供丰富的查询功能
        • 支持全文搜索、模糊查询、分词查询、聚合分析等高级特性
        • 提供强大的DSL查询语言和SQL接口
        • 内置倒排索引，亿级数据下仍能保持秒级响应
      • 典型场景：HBase适合"已知RowKey查明细”，ES适合"根据内容特征找文档"
    • 性能与扩展性
      • 写入性能
        • 两者都基于LSM树结构，理论写入性能相当
        • 实际场景中HBase写入更高效，因为ES需要构建倒排索引等额外开销
        • HBase单集群可支持每天TB级数据写入
      • 查询性能
        • ES在复杂查询和全文检索方面性能显著优于HBase
        • HBase在简单键值查询和范围扫描上延迟更低
      • 扩展能力
        • HBase天生为海量数据设计，通过Region分裂实现水平扩展
        • ES通过分片(Shard)机制扩展，但数据量极大时可能需要业务层干预
    • 存储成本与资源消耗
      • 存储效率
        • HBase采用列式存储+压缩，存储效率更高，适合长期保存海量数据
        • ES因维护索引和副本，存储开销通常比HBase高30%-50%
      • 资源消耗
        • ES的CPU和内存消耗更高（分词、索引构建等）
        • HBase更注重I/O优化，适合资源受限环境
    • 维护复杂度
      • HBase依赖完整的Hadoop生态(HDFS/ZooKeeper等)，组件多，运维复杂度高
      • Elasticsearch自成体系，部署和维护相对简单，但集群调优需要专业知识
    • 实时性与一致性
      • HBase采用强一致性模型(基于2PC协议)，适合对一致性要求高的场景
      • Elasticsearch采用最终一致性，查询实时性更好但可能短暂不一致
    • 典型应用场景推荐
      • 适合选择HBase的场景
        • 海量时序数据存储：如设备监控数据、日志流水(每日5G以上)
        • 高吞吐写入：需要持续写入TB级结构化数据的场景
        • 简单查询为主：已知主键的快速查询或固定模式的范围扫描
        • 长期数据归档：需要低成本长期保存的历史数据
        • 案例：某埋点系统每天100G数据通过Kafka写入HBase，稳定运行
      • 适合选择Elasticsearch的场景
        • 全文检索需求：如商品搜索、内容检索、日志分析
        • 复杂查询分析：需要多条件组合、聚合统计的业务
        • 实时数据分析：如应用监控、安全分析等需要秒级响应的场景
        • 灵活的数据结构：字段频繁变化或无法预定义schema的情况
      • 混合架构方案
        • 对于同时需要海量存储和复杂查询的场景，可考虑ES+HBase组合架构
          • HBase作为主存储，保证数据持久性和高吞吐写入
          • ES作为索引层，提供丰富的查询能力
          • 通过数据同步工具保持两者数据一致
        • 百亿级数据架构案例：使用ES处理复杂查询，HBase保证可靠存储，两者协同工作
    • 选型决策关键因素
      • 数据规模：日增5GB以下ES更简单，TB级选HBase
      • 查询复杂度：简单查询HBase足够，复杂搜索选ES
      • 数据结构：固定结构用HBase，多变结构用ES
      • 资源预算：资源有限优先HBase，可接受较高成本选ES
      • 团队技能：熟悉Hadoop选HBase，无大数据经验ES更易上手
    • 总结建议
      • Elasticsearch是搜索和复杂查询的首选，特别适合日志分析、内容检索等需要灵活查询的场景
      • HBase是海量数据存储的可靠选择，特别适合时序数据、用户行为记录等高吞吐写入场景

    HBase与RDBMS的区别#

    • 存储方式：HBase为列式存储，RDBMS为行式存储
    • 扩展性：HBase支持水平扩展，RDBMS通常垂直扩展
    • 事务：HBase仅支持单行事务，RDBMS支持多行复杂事务

    HBase与Hive的区别#

    • 用途：HBase是实时数据库，Hive是离线数据仓库
    • 查询性能：HBase支持毫秒级查询，Hive依赖MapReduce，延迟高

    数据存储结构#

    • Region：表的水平分区，按RowKey范围划分
    • MemStore：写缓存，满时刷盘生成HFile
    • HFile：底层存储文件，基于HDFS的列族数据块

    hbase 的特点#

    • 易扩展
    • 海量存储
    • 列式存储
    • 高可靠性
    • 稀疏性

    模块组成#

    • HBase 可以将数据存储在本地文件系统，也可以存储在 HDFS 文件系统。在生产环境中，HBase 一般运行在 HDFS 上，以 HDFS 作为基础的存储设施。HBase 通过 HBase Client 提供的 Java API 来访问 HBase 数据库，以完成数据的写入和读取。HBase 集群主由HMaster、Region Server 和 ZooKeeper 组成
      • HMaster
        • 负责管理 RegionServer，实现其负载均衡
        • 管理和分配 Region，比如在 Region split时分配新的 Region，在 RegionServer 退出时迁移其内的 Region 到其他 RegionServer上
        • 管理namespace和table的元数据（实际存储在HDFS上）
        • 权限控制（ACL）
      • RegionServer
        • 存放和管理本地 Region
        • 读写HDFS，管理Table中的数据
        • Client 从 HMaster 中获取元数据，找到 RowKey 所在的 RegionServer 进行读写数据
      • ZooKeeper
        • 存放整个 HBase集群的元数据以及集群的状态信息
        • 实现HMaster主从节点的failover

    RowKey设计原则#

    • 长度：建议≤16字节，避免内存和存储浪费
    • 散列性：高位散列（如哈希前缀）防止热点问题
    • 唯一性：必须保证RowKey唯一标识数据

    应用场景#

    • 适用场景
      • 半结构化/非结构化数据（如日志、用户行为）
      • 高并发实时读写（如电商订单、社交消息）
      • HBase是根据谷歌的BigTable设计的。典型的应用场景就是不断插入新的信息（谷歌的情况下就是互联网上新生成的网页），而不怎么修改。比如现在Facebook的messenger就是用HBase实现的
      • Facebook
        • Facebook用HBase存储在线消息，每天数据量近百亿，每月数据量250 ~ 300T， HBase读写比基本在1:1，吞吐量150w qps
      • 小米
        • 10+在线HBase集群，好几百台服务器，米聊历史数据，消息push系统等多个重要应用系统都建立在HBase基础之上
      • 网易
        • 哨兵监控系统，云信历史数据，日志归档数据等一系列重要应用底层都由HBase提供服务
      • 写密集型应用，每天写入量巨大，而相对读数量较小的应用，比如IM的历史消息，游戏的日志等等
      • 不需要复杂查询条件来查询数据的应用，HBase只支持基于rowkey的查询，对于HBase来说，单条记录或者小范围的查询是可以接受的，大范围的查询由于分布式的原因，可能在性能上有点影响，而对于像SQL的join等查询，HBase无法支持
      • 对性能和可靠性要求非常高的应用，由于HBase本身没有单点故障，可用性非常高
      • 数据量较大，而且增长量无法预估的应用，HBase支持在线扩展，即使在一段时间内数据量呈井喷式增长，也可以通过HBase横向扩展来满足功能
    • 不适用场景
      • 需要复杂事务或多表关联查询的业务
