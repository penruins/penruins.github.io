    分区表的索引如何创建，请以postgresql举例 #

    PostgreSQL分区表索引创建指南 #

    PostgreSQL中分区表索引的创建方法与普通表有所不同，需要特别注意分区表的特性。以下是详细的创建方法和示例：

    一、基本索引创建方法 #

    1. 自动分区索引（PostgreSQL 11+） #

    从PostgreSQL 11开始，可以直接在父表上创建索引，系统会自动为所有子分区创建对应的索引1247。

    -- 创建分区表
CREATE TABLE sales (
    id SERIAL PRIMARY KEY,
    product VARCHAR,
    sale_date DATE
) PARTITION BY RANGE (sale_date);

-- 创建子分区
CREATE TABLE sales_2021_01_01 PARTITION OF sales FOR VALUES FROM ('2021-01-01') TO ('2021-01-02');
CREATE TABLE sales_2021_01_02 PARTITION OF sales FOR VALUES FROM ('2021-01-02') TO ('2021-01-03');

-- 在父表上创建索引，会自动应用到所有分区
CREATE INDEX idx_product ON sales (product);

    2. 手动创建分区索引（PostgreSQL 10及以下） #

    在PostgreSQL 10及以下版本，需要为每个分区单独创建索引：

    -- 为每个分区单独创建索引
CREATE INDEX sales_2021_01_01_product_idx ON sales_2021_01_01 (product);
CREATE INDEX sales_2021_01_02_product_idx ON sales_2021_01_02 (product);

    二、特殊索引类型 #

    1. 唯一索引和主键 #

    PostgreSQL 11+支持在分区表上创建唯一约束和主键，但必须包含分区键47：

    -- 创建包含分区键的唯一索引
CREATE TABLE rtable(c1 INT, c2 VARCHAR(10)) PARTITION BY RANGE(c1);
ALTER TABLE rtable ADD CONSTRAINT pk_rtable PRIMARY KEY(c1);

    2. 局部索引(Partial Index) #

    可以只为满足特定条件的行创建索引，节省空间并提高查询效率25：

    -- 只为活跃客户创建索引
CREATE INDEX idx_customer_inactive ON customer(active) WHERE active = 0;

    3. 全局索引(Global Index) #

    PostgreSQL支持全局索引，可以跨分区查询而不需要包含分区键2743：

    -- 创建全局索引
CREATE INDEX idx_global ON employees(email) GLOBAL;

    三、在线创建索引的注意事项 #

    PostgreSQL不支持直接在分区表上并发创建索引(CONCURRENTLY)，但可以通过以下方法实现类似效果17：

    1. 在每个分区上并发创建索引
    2. 在父表上以非并发方式创建索引
    -- 错误方法（会报错）
CREATE INDEX CONCURRENTLY idx_product ON sales (product);

-- 正确方法
-- 1. 在每个分区上并发创建索引
CREATE INDEX CONCURRENTLY sales_2021_01_01_product_idx ON sales_2021_01_01 (product);
CREATE INDEX CONCURRENTLY sales_2021_01_02_product_idx ON sales_2021_01_02 (product);

-- 2. 在父表上创建索引
CREATE INDEX idx_product ON sales (product);

    四、索引类型选择 #

    PostgreSQL支持多种索引类型，应根据查询需求选择合适的类型4041：

    1. B-tree：默认索引类型，适合范围查询和等值查询
    2. Hash：只支持等值查询，但性能更好
    3. GIN：适合全文搜索和数组查询
    4. GiST：适合空间数据和复杂数据类型
    5. BRIN：适合大型有序数据集，占用空间小
    -- 创建GIN索引示例
CREATE INDEX idx_gin ON documents USING gin (content gin_trgm_ops);

    五、最佳实践建议 #

    1. 索引应包含查询条件中的常用列1
    2. 对于分区表，考虑查询是否使用分区键来决定索引类型27
    3. 定期分析索引使用情况，删除不必要的索引
    4. 对于大型分区表，考虑使用BRIN索引减少空间占用40
    5. 唯一约束必须包含分区键列47

    通过合理设计分区表索引，可以显著提高PostgreSQL数据库的查询性能和管理效率。
