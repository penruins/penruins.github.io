    database #

    索引 index #

    什么是聚集索引、非聚集索引？ #

    • 聚集索引：数据行的物理存储顺序与索引的逻辑顺序一致
    • 非聚集索引：数据行的物理存储顺序与索引的逻辑顺序不一致

    为什么只能有一种聚集索引？ #

    • 因为数据物理的存储顺序只能有一种，因此只能有一种聚集索引

    默认的聚集索引是什么？ #

    • 主键

    什么是回表？ #

    • 当查询条件使用普通索引并且查询的列不包含在索引中，就需要查询完普通索引后再去查询聚集索引的树拿到完整的行数据

    当底层建了abc的联合索引，底层会建a,ab, abc这3个索引么？ #

    • 不会，只创建一个联合索引树

    联合索引（abc)，当查询ab, ac会走索引么？ #

    • ab 走索引
    • ac 只走 a 索引

    order by 和 group by 如何创建索引？ #

    • order by
      • 单列 直接创建
      • 多列 创建联合索引, 索引顺序要和 order by 的顺序一致
        • 不一致索引会失效
    • group by
      • group by 实际上是先进行排序，再进行分组。所以遵循 order by 的索引机制

    MySQL索引失效的20种场景 #

    • MySQL索引失效的20种场景
    • 1 查询条件未使用索引列
      • SELECT * FROM users WHERE non_indexed_column = 'example'
    • 2 查询条件对索引列使用了函数或表达式
      • SELECT * FROM users WHERE LEFT(name, 3) = 'Ali'
    • 3 数据类型不一致引发隐式类型转换
      • SELECT * FROM users WHERE phone = 1234567890; -- phone 是 VARCHAR
    • 4 联合索引使用时的范围查询阻断
      • SELECT * FROM users WHERE name = 'Alice' AND age > 30 AND city = 'New York'
    • 5 如果 OR 条件中有字段未使用索引，会导致整体索引失效
      • SELECT * FROM users WHERE name = 'Alice' OR age = 30;
      • 解决方法：拆分查询，用 UNION 替代
        • SELECT * FROM users WHERE name = 'Alice' UNION SELECT * FROM users WHERE age = 30;
    • 6 LIKE 查询中使用前置通配符 % 时，无法利用索引
      • SELECT * FROM users WHERE name LIKE '%Alice';
      • 解决方法
        • 替换为后置通配符 LIKE ‘Alice%’
        • 使用全文索引
          • ALTER TABLE users ADD FULLTEXT(name);
          • SELECT * FROM users WHERE MATCH(name) AGAINST('Alice');
    • 7 不等值操作符 != <> 通常会导致索引失效
      • SELECT * FROM users WHERE age != 30;
      • 解决方法：改写为范围查询
        • SELECT * FROM users WHERE age < 30 OR age > 30;
    • 8 IN 子句中参数过多时，索引可能失效
      • SELECT * FROM users WHERE id IN (1, 2, ..., 10000)
      • 解决方法：分批查询或控制 IN 参数数量
        • SELECT * FROM users WHERE id BETWEEN 1 AND 5000
    • 9 某些情况下对NULL值的查询可能导致索引失效
      • SELECT * FROM users WHERE email IS NULL;
      • 解决方法：避免使用 NULL，改为特定占位值
    • 10 ORDER BY 和 GROUP BY 不符合索引顺序
      • select * from users order by age, name; 索引顺序为(name,age)
      • 解决方法：调整查询或索引顺序一致
    • 11 索引列未覆盖查询字段
      • select age from users where name = 'Alice'
      • 解决方法：创建覆盖索引
        • alter table users add index(name, age)
    • 12 表统计信息过期
      • 表的统计信息不准确时，MySQL优化器可能错误选择全表扫描
      • 解决方法：定期更新统计信息
        • ANALYZE TABLE users
    • 13 数据分布不均
      • select * from users where gender = 'M' -- gender 列 90% 的值相同
      • 解决方法：优化查询字段或调整数据分布
    • 14 子查询未优化
      • 子查询结果集过大时，索引失效
      • SELECT * FROM users WHERE id IN (SELECT user_id FROM orders WHERE total > 100)
      • 解决方法：改为 JOIN 查询
        • SELECT u.* FROM users u JOIN orders o ON u.id = o.user_id WHERE o.total > 100
    • 15 锁机制干扰 间隙锁
      • 描述：间隙锁可能阻止索引的高效使用
      • SELECT * FROM users WHERE id BETWEEN 10 AND 20 FOR UPDATE
      • 解决方法：降低事务隔离级别至 Read Committed(读已提交)
        • 可重复读加上间隙锁防止幻读
    • 16 索引被动态调整
      • MySQL优化器动态调整索引，可能选择非最佳索引或直接全表扫描
      • SELECT * FROM users FORCE INDEX (idx_name) WHERE name = 'Alice'
    • 17 聚合函数未走索引
      • 使用聚合函数(如 COUNT, SUM) 时，索引可能无法被利用
      • SELECT COUNT(*) FROM users WHERE age > 30
      • 解决方法：利用覆盖索引
        • ALTER TABLE users ADD INDEX(age)
    • 18 使用了非等值JOIN
      • select * from users u join orders o on u.id > o.user_id
      • 优化连接条件，尽量使用等值连接
    • 19 小表全表扫描替代索引扫描
      • 对于非常小的表，MySQL优化器可能选择全表扫描而非索引
      • SELECT * FROM small_table WHERE id = 1;
      • 解决方法：无需优化，小表全表扫描的代价通常可以忽略
    • 20 分区表查询不走全局索引
      • 分区表的查询条件不覆盖分区键时，全局索引可能无法生效
      • SELECT * FROM partitioned_users WHERE city = 'New York'
      • 解决方法：在分区键上创建局部索引

    分区 #

    分表和分区在物理存储上有什么区别？ #

    • 在数据库中一张表对应磁盘中的2个文件，拓展名为idb的文件和拓展名为frm的文件，idb文件存储表中的记录，frm存储表的结构信息
    • 进行分表的话有几张表就有几个对应的文件，进行分区就只有一个frm文件和若干个idb文件，一个分区对应一个idb文件
    • 分区能提升大表的读写性能就是因为根据分区规则仅仅扫描对应分区的磁盘文件，而不用进行全表扫描

    一张表最多可以分多少个区？ #

    • 1024

    事务 #

    数据隔离级别 #

    • 读未提交 Read Uncommitted
    • 读已提交 Read Committed
      • 脏读：同一个事务中，多次读取同一数据的结果不同
    • 可重复读 Repeatable Read
      • 幻读：同一个事务中，相同的查询条件读取到的数据条数不一样
    • 串行化 Serializable

    MVCC #

    MVCC 的全称是什么？ #

    • Multi-Version Concurrency Control 多版本并发控制

    MySQL的事务引擎仅仅使用行级锁吗？ #

    • 不是，它结合了行级锁+MVCC(多版本并发控制)

    MVCC适用于哪些隔离级别？ #

    • 适用于 REPEATED READ 和 READ COMMITTED
    • 不适用于 READ UNCOMMITTED
