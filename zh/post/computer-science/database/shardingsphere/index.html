    概览 #

    什么是 shardingsphere #

    • Apache ShardingSphere 是一款分布式的数据库生态系统， 可以将任意数据库转换为分布式数据库，并通过数据分片、弹性伸缩、加密等能力对原有数据库进行增强
    • Apache ShardingSphere 设计哲学为 Database Plus，旨在构建异构数据库上层的标准和生态。 它关注如何充分合理地利用数据库的计算和存储能力，而并非实现一个全新的数据库。 它站在数据库的上层视角，关注它们之间的协作多于数据库自身
    • ShardingSphere-JDBC
      • ShardingSphere-JDBC 定位为轻量级 Java 框架，在 Java 的 JDBC 层提供的额外服务
    • ShardingSphere-Proxy
      • ShardingSphere-Proxy 定位为透明化的数据库代理端，通过实现数据库二进制协议，对异构语言提供支持

    高性能数据库集群 #

    第一种：读写分离 #

    • 主库负责处理事务性的增删改操作，从库负责处理查询操作
    • 能够有效的避免由数据更新导致的行锁，使得整个系统的查询性能得到极大的改善
    • 读写分离是根据SQL语义的分析，将读操作和写操作分别路由至主库与从库
    • 两种方式
      • 一主多从
      • 多主多从

    第二种：数据库分片 #

    • 读写分离分散了数据库读写操作的压力，但是没有分散存储的压力，为了满足业务数据存储的需求，就需要将存储分散到多台数据库服务器上
    • 数据分片的有效手段是对关系型数据库进行分库和分表
    • 数据分片的拆分方式
      • 垂直分片
        • 专库专用
          • 例如将商城库拆分为订单库、用户库、库存库
        • 垂直拆分可以缓解数据量和访问量带来的问题，但无法根治。如果垂直拆分之后，表中的数据量依然超过单节点所能承载的阈值，则需要水平分片来进一步处理
        • 垂直分表适合将表中某些不常用的列，或者是占了大量空间的列拆分出去
      • 水平分片
        • 水平分表
          • 单表切分为多表之后，新的表即使在同一个数据库服务器中，也可能带来可观的性能提升，如果性能能够满足业务要求，可以不拆分到多台数据库服务器，毕竟业务分库也会引入很多复杂性
        • 水平分库
          • 如果单表拆分为多表后，单台服务器依然无法满足性能要求，那就需要将多个表分散在不同的数据库服务器中
    • CAP理论中的C在实践中是不可能完美实现的，在数据复制的过程中，节点N1和节点N2的数据并不一致（强一致性）。即使无法做到强一致性，但应用可以采用合适的方式达到最终一致性
      • 基本可用
      • 软引用
      • 最终一致性
    • 阿里巴巴Java开发手册
      • 单表行数超过500万行或者单表容量超过2GB，才推荐进行分库分表
      • 如果预计3年后的数据量根本达不到这个级别，请不要在创建表时就分库分表
    • 读写分离和数据分片具体的视线方式有两种
      • 程序代码封装
        • 在代码中抽象一个数据访问层（中间层封装），实现读写操作分离和数据库服务器连接的管理
        • Apache ShardingSphere
          • ShardingSphere-JDBC 程序代码封装
          • ShardingSphere-Proxy 中间件封装
      • 中间件封装
        • 独立一套系统处理，实现读写操作分离和数据库服务器连接的管理。对于业务服务器来说，访问中间件和访问数据库没有区别，在业务服务器看来，中间件就是一个数据库服务器
        • MyCat

    主从同步原理 #

    • slave会从master读取binlog来进行数据同步
    • 1 master将数据改变记录到bin log中
    • 2 当 slave 上执行 start slave 命令之后，slave会创建一个IO连接用来连接master，请求master中的binlog
    • 3 当 slave 连接 master 时，master 会创建一个 log dump 线程，用于发送 bin log 的内容。在读取 binlog 的内容的操作中，会对主节点中的binlog加锁，当读取完成并发送给从服务器后解锁
    • 4 IO 线程接收主节点 binlog dump 进程发来的更新之后，保存到中继日志 relay log 中
    • 5 slave 的 SQL 线程，读取 relay log 日志，并解析成具体操作，从而实现主从操作一直，最终数据一致

    MyCat与ShardingSphere深度对比 #

    • 如果是从传统单体结构迁移，且只需要基本的分库分表功能，MyCat可能更合适
    • 如果是新建的云原生系统或微服务架构，ShardingSphere是更好的选择
    • 考虑到技术演进和社区支持，新项目建议优先考虑ShardingSphere
    • 某些场景可以组合使用-用ShardingSphere-JDBC处理应用内分片，用MyCat/ShardingShpere-Proxy处理遗留系统访问

    ShardingSphere的分布式事务功能具体是如何实现的？ #

    XA事务（强一致性） #

    • 实现原理
      • 基于XA协议的两阶段提交(2PC)
      • 使用ShardingSphere的XATransactionManager作为事务管理器
    • 工作流程
      • 准备阶段
        • 事务协调器向所有参与者发送prepare请求
        • 各参与者执行事务但不提交，记录undo/redo日志
        • 参与者返回准备就绪或失败响应
      • 提交/回滚阶段
        • 如果所有参与者都准备就绪，发送commit请求
        • 如果任一参与者准备失败，发送rollback请求
        • 参与者根据指令完成最终操作

    Seata事务（柔性事务/AT模式） #

    • 整合Seata的AT(Automatic Transaction)模式
    • 基于全局锁+分支事务的补偿机制
    • 工作流程
      • 第一阶段
        • 业务SQL被Seata拦截解析
        • RM生成undo log记录修改前数据
        • 执行业务SQL并提交本地事务
        • 向TC注册分支事务并报告状态
      • 第二阶段
        • 成功时：异步删除undo log
        • 失败时：基于 undo log 进行补偿回滚

    BASE事务（Saga模式） #

    • 基于Saga模式的长事务解决方案
    • 每个子事务都有对应的补偿操作
    • 最终一致性模型
    • 执行模式
      • 正向流程：按顺序执行所有子事务
      • 补偿流程：出现失败时，逆序执行补偿操作

    生产实践建议 #

    • 金融核心系统：优先考虑XA事务
    • 普通交易系统：推荐Seata AT模式
    • 跨系统长流程：考虑Saga模式
